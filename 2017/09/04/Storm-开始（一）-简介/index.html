Storm简介

标签（空格分隔）： Storm 实时流处理

1. 什么是Storm

  1. Storm是一个分布式计算框架，主要使用Clojure与Java语言编写，最初由Nathan Marz带领BackType公司的团队创建，在Backtype公司被Twitter公司收购够进行开源；
  2. Storm能轻松可靠的处理无界的数据流，就像Hadoop对数据进行批处理；
  3. Storm能用到很多处理场景中，包括实时分析、在线机器学习、连续计算、分布式RPC、ETL等；
  4. Storm 的处理速度非常快，每个节点每秒中可以处理100万条消息。同时，Storm是可伸缩的、容错的，并且能保证数据根据用户设定被妥善处理，便于进行设置和操作。
  5. Storm 集成了许多消息队列和数据库技术。Storm的Topology消耗数据流，以任意复杂的方式处理这些流，并且在每个需要计算的阶段以这些流进行重新划分。

2. Storm核心技术和基本组成

  1. Topology
    一个拓扑是一个图的计算。用户在一个拓扑的每个节点包含处理逻辑，节点之间的链接显示数据应该如何在节点之间传递。
  2. Stream
    流式Storm 的核心抽象。一个流式一个无界的Tuple序列，Tuple可以包含整型、长整型、短整型、字节、字符、双精度数、浮点数、布尔值和字节数组。用户可以定义序列化器在本机Tuple使用自定义类型。
  3. Spout
    Spout是Topology流的来源，一般Spout以外部来源读取Tuple，提交到Topology。Spout可以发出超过一个流。
  4. Bolt
    Topology中的所有处理都在Bolt中完成。Bolt可以完成过滤、业务处理、链接运算、连接、访问数据库等业务。可以完成流的转换，发出超过一个流，主要方法是execute方法。Bolt可以启动新线程做异步处理。
  5. Stream grouping
    流分组用于在Bolt的任务中定义流应该如何分区。有8个内置的流分组接口：随机（shuffle）、字段（Fields）、部分关键字（Partial key）、全部（All）、全局（Global）、无分组（None）、直接（Direct）、本地或随机（Local or shuffle）分组
  6. Task
    每个Spout或Bolt在集群执行许多任务。每个任务对应一个线程执行，流分组定义如何从一个任务集到另一个任务集发送Tuple。通过TopologyBuilder类的setSpout()和setBolt()方法设置每个Spout或Bolt的并行度。
  7. Worker
    Topology跨一个或多个worker节点的进程执行。每个worker节点的进程是一个物理的jvm和Topology执行所有任务的一个子集。

3. Storm的特点

  1. 集成多种技术
    • Storm集成了一些消息队列系统，如Kestrel、RabbitMQ、Kafka、JMS等；
    • Storm也可以很容易的数据库系统集成，Storm会自动完成并行化、分区、失败重连等操作。
  2. 简单的API
    • 当使用Storm编程时，可以使用相应的API来操纵和变换Tuple的流。一个Tuple是一个命名的值列表。Tuple可以包含任何类型的对象，还可通过序列化注册新类型。
    • Storm只包含三个抽象类型：Spout、Bolt、Topology
  3. 可拓展
    • Storm的Topology是并行计算的，运行在一个集群主机上，可以对Topology的不同部分单独调整他们的并行扩展。
  4. 容错
    • 当worker死亡，Storm会自动重新启动它们。如果一个节点死亡，worker将在另一个节点重新启动。
    • Storm的守护进程Nimbus和Supervisor被设计为无状态和快速失败的，如果挂掉重启后像是什么都没发生过一样。
  5. 保证数据处理
    • 通过跟踪Topology中的Tuple确保每一个Tuple都能被完全处理。
  6. 多语言支持
    • Storm的核心是一个定义和提交Topology的Thrift定义，所以可以使用任何语言定义。
    • 非jvm的Spout和Bolt可以通过一个基于JSON协议的stdin/stdout与Storm通信。
  7. 自由开源
  8. 部署操作简单

4. Storm应用方向

  1. 流处理
    • 处理源源不断流进来的消息，处理之后将结果写入到某个存储中去。
  2. 连续计算
    • Storm能保证计算可以永远运行，知道用户结束计算进程为止。
  3. 分布式RPC
    • 由于Storm的处理组件是分布式的，而且处理延迟极低，所以可以作为通用的分布式RPC框架使用。
